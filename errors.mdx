The API uses standard HTTP status codes and returns structured error objects in the response body.

## Error format

```json
{
  "error": {
    "type": "validation_error",
    "code": "BOOKING_SLOT_UNAVAILABLE",
    "message": "The requested time slot is no longer available.",
    "param": "slot_id",
    "request_id": "req_abc123",
    "doc_url": "https://docs.platform.io/errors/BOOKING_SLOT_UNAVAILABLE"
  }
}
```

| Field | Description |
|-------|-------------|
| `type` | Error category (see below) |
| `code` | Machine-readable error code |
| `message` | Human-readable description |
| `param` | The request parameter that caused the error (when applicable) |
| `request_id` | Unique identifier for this request — include when contacting support |
| `doc_url` | Link to documentation about this specific error |

## Error types

| Type | Description | Retry? |
|------|-------------|--------|
| `api_error` | Server-side issue | Yes, with backoff |
| `validation_error` | Invalid request parameters | No — fix the request first |
| `business_error` | Valid request but business rules prevent it | Depends on the rule |
| `idempotency_error` | Idempotency key reused with different parameters | No — use a new key |

## HTTP status codes

| Status | Meaning |
|--------|---------|
| `200` | Request succeeded |
| `201` | Resource created |
| `400` | Bad request — check `param` field |
| `401` | Authentication failed or missing |
| `403` | Insufficient permissions |
| `404` | Resource not found |
| `409` | Conflict — idempotency key reuse or concurrent modification |
| `422` | Validation error — request body is well-formed but contains invalid values |
| `429` | Rate limit exceeded — retry after the `Retry-After` header |
| `500` | Internal server error — retry with backoff |

## Handling errors

<Tabs>
  <Tab title="cURL">
  ```bash
  curl -s -w "\n%{http_code}" https://api.platform.io/v1/customers/cus_invalid \
    -H "Authorization: Bearer sk_test_your_key_here"
  ```
  </Tab>
  <Tab title="Node.js">
  ```javascript
  const response = await fetch("https://api.platform.io/v1/customers/cus_invalid", {
    headers: { "Authorization": "Bearer sk_test_your_key_here" }
  });

  if (!response.ok) {
    const { error } = await response.json();
    console.error(`${error.type}: ${error.code} — ${error.message}`);
  }
  ```
  </Tab>
  <Tab title="Python">
  ```python
  response = requests.get(
      "https://api.platform.io/v1/customers/cus_invalid",
      headers={"Authorization": "Bearer sk_test_your_key_here"}
  )

  if response.status_code >= 400:
      error = response.json()["error"]
      print(f"{error['type']}: {error['code']} — {error['message']}")
  ```
  </Tab>
</Tabs>

## Request IDs

Every response includes a `Request-Id` header. Log these and include them when contacting support to speed up debugging.
